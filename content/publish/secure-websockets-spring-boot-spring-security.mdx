## How does JWT authentication work with WebSockets?

In this implementation, **JWT (JSON Web Token)** authentication is used to secure both the REST API and the WebSocket connections. The flow works as follows:

1.  The client authenticates via the REST API login endpoint and receives a **JWT token**
2.  When establishing a WebSocket connection, the client includes the JWT token as a **STOMP header** on the CONNECT frame
3.  The **WebSocketTokenInterceptor** intercepts the CONNECT frame, extracts the JWT token, validates it, and establishes the **SecurityContext** for the connection
4.  All subsequent SUBSCRIBE and MESSAGE frames are authenticated against the established security context

This approach using **jjwt 0.12.6** replaces the custom HMAC token mechanism from the original implementation, providing a standards-based, stateless authentication flow.

## Spring Security 6.4

**Spring Security 6.4** provides the foundation for securing both the REST endpoints and WebSocket connections. The modern approach uses **SecurityFilterChain** beans instead of the legacy WebSecurityConfigurerAdapter.

### What is supported out of the box

Spring Security by default has security for WebSockets and can use the message header simpUser to authenticate, but when you want to authenticate on a different identifier — in this example a JWT token — then custom configuration is required.

### How JWT validation works on CONNECT

The **WebSocketTokenInterceptor** implements `ChannelInterceptor` and intercepts inbound STOMP messages. On a **CONNECT** frame, it:

1.  Extracts the JWT token from the STOMP native headers
2.  Validates the token using the **jjwt** library
3.  Loads the user details and creates an **Authentication** object
4.  Sets the **SecurityContext** for the WebSocket session

This means that any **CONNECT**, **SUBSCRIBE**, or **MESSAGE** frame requires a valid JWT token to be passed as part of the STOMP headers, otherwise the response will be AccessDenied. **UNSUBSCRIBE** and **DISCONNECT** frames are permitted without authentication as headers cannot be attached to these frame types.

### Additional security features

*   **BCrypt password encoding** for secure password storage
*   **CORS configuration** to allow the Angular development server to communicate with the API
*   **Stateless session management** — no server-side HTTP sessions
*   **Method-level security** with @PreAuthorize annotations

## STOMP Message-Level Security

**STOMP (Simple Text Oriented Messaging Protocol)** provides a frame-based protocol on top of WebSockets, allowing for structured messaging with headers, destinations, and subscriptions.

The security rules are configured so that:

*   **CONNECT** — requires a valid JWT token in the STOMP headers
*   **SUBSCRIBE** — requires authentication (user must have connected with a valid token)
*   **MESSAGE** — requires authentication
*   **UNSUBSCRIBE** — permitted for all (no headers can be attached)
*   **DISCONNECT** — permitted for all (no headers can be attached)
*   **All other messages** — denied by default

This provides a layered security model where the REST API is secured by Spring Security's HTTP filter chain, and the WebSocket connections are additionally secured at the STOMP message level.
