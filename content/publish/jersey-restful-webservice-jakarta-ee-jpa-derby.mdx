RESTful is a key term that remains central to modern API development.

Previous services used to be created implementing JAX-WS with the communication protocol being SOAP, various implementations were and are still used but from my own experience they all have fallen short especially in the area of code generation, these technologies included Axis 1 & 2, Apache CXF as well as the java generation tools.

Since RESTful implements **Jakarta RESTful Web Services** (formerly JAX-RS) with the communication protocol being HTTP, it has allowed for rapid development with the ability to send data in different formats, typically it is JSON however it can also be sent as XML or even plain text.

This project has been fully modernised to use **Jersey 4.0.0**, **Jakarta EE 11** (the `jakarta.*` namespace), **Java 21**, and modern testing with **JUnit 5** and **Mockito 5.6**. The build includes **CI/CD with GitHub Actions** for automated testing on every push.

## Jersey 4.0.0

### Why choose this technology

For this example I wanted to have a lightweight RESTful service that could perform functions/operations and return the appropriate response to the calling client.

**Jersey 4.0.0** is the reference implementation for **Jakarta RESTful Web Services** under **Jakarta EE 11**. It is a lightweight framework that allows for simple and rapid development of RESTful web services, and can easily be integrated with many other technologies to add additional bells and whistles.

The major change from earlier versions is the migration from the `javax.*` namespace to the `jakarta.*` namespace, reflecting the transition from Java EE to Jakarta EE.

### What is currently used

*   **Jakarta Servlets** with the `jakarta.*` namespace
*   RESTful interface via **Jakarta RESTful Web Services**
*   HTTP protocol communication
*   Custom Application Configuration using **ResourceConfig**
*   **HK2** dependency injection
*   Annotations throughout
*   **JUnit 5.10** and **Mockito 5.6** testing framework

### What can you add

*   HTTP methods that are not being used but make sense to enhance existing functionality these include DELETE, PUT etc.
*   Validation of data received in the RESTful interface using **Jakarta Bean Validation**.

## Jackson (via jersey-media-json-jackson)

### Why choose this technology

For this example I was required to use data binding e.g. JSON to Java and back.

**Jackson** is integrated via the **jersey-media-json-jackson** module, which provides seamless JSON data binding for Jersey 4. Jackson is the industry-standard JSON library for Java and provides powerful serialisation and deserialisation capabilities out of the box.

This replaces the previously used Genson library, aligning the project with the most widely adopted JSON binding solution in the Java ecosystem.

### What is currently used

*   **Automatic JSON data binding** via Jackson
*   Configured through the Jersey Jackson module — no manual POJO mapping required

## EclipseLink 4.0.2

### Why choose this technology

For the example I found it important to have the data stored like you would in the industry.

I am a big fan of Object Relational Mapping and wanted to utilise **Jakarta Persistence 3.1**.

**EclipseLink 4.0.2** is the reference implementation of Jakarta Persistence 3.1, providing full support for the modern JPA specification under the `jakarta.persistence.*` namespace. While you can use many other technologies I found this one was still active, lightweight and had the support required.

### What is currently used

*   **Jakarta Persistence 3.1** support
*   Annotations are used and encouraged to configure your Entities using `jakarta.persistence.*`
*   Persistence configuration via **persistence.xml**
*   Entity-to-table mapping
*   **EntityManager** and transaction management

### What can you add

*   Additional unit tests for the DAO layer and entities using JUnit 5.

## Derby 10.16

### Why choose this technology

For the example I wanted to have a simple in-memory database that did not require additional configuration once this demo is checked out.

**Derby 10.16** has great support for an in-memory relational database and is simple to configure. It remains a solid choice for development and testing scenarios where you want zero external database setup.

### What is currently used

*   **In-memory database** — no external database installation required
*   Automatic table creation via JPA schema generation
*   Compatible with Jakarta Persistence 3.1

### What can you add

*   Unit tests with pre-populated data for detailed DAO tests.
