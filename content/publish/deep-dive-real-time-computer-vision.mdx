## What is Computer Vision?

Computer vision is the field of artificial intelligence that enables machines to interpret and make decisions based on visual data — images and video. At its core, it answers a deceptively simple question: *what is in this image, and where is it?*

The evolution of computer vision has followed two distinct eras:

**Traditional Computer Vision** relied on hand-crafted feature extraction. Engineers would write explicit rules to detect edges (Canny, Sobel), extract features (SIFT, SURF, HOG), and feed those features into classical machine learning classifiers like Support Vector Machines or Random Forests. This approach worked well for constrained environments — think barcode scanning or simple shape detection — but was brittle. Changes in lighting, angle, or background could break the entire pipeline.

**Deep Learning Computer Vision** replaced manual feature engineering with learned representations. Convolutional Neural Networks (CNNs) like ResNet, YOLO, and EfficientNet learn to extract features directly from raw pixels during training. More recently, Vision Transformers (ViT) have brought the attention mechanism from natural language processing into the visual domain. The result is systems that are significantly more robust to real-world variation and can handle complex tasks like multi-object detection, instance segmentation, pose estimation, and face recognition simultaneously.

![Traditional CV vs Deep Learning CV](/blog/deep-dive-real-time-computer-vision/inline-1.png)

In a real-time computer vision platform, we are primarily concerned with **inference** — running trained models against live video streams to produce actionable detections. The challenge is not just accuracy; it is doing so at 25–30 frames per second across dozens or hundreds of concurrent camera streams with sub-100ms latency. That is where the architecture becomes critical.

## Terminology

Before we go deeper, let us define the key terms you will encounter throughout this post:

**Inference** — The process of running a trained model against new input data to produce predictions. In computer vision, this means passing video frames through a neural network to detect objects, classify scenes, or extract features.

**RTSP (Real-Time Streaming Protocol)** — A network protocol designed for controlling streaming media servers. IP cameras typically expose an RTSP endpoint (e.g., `rtsp://camera-ip:554/stream1`) that clients connect to for live video.

**ONVIF (Open Network Video Interface Forum)** — An industry standard for IP-based security products. ONVIF provides a common protocol for camera discovery, configuration, and stream access, allowing platforms to automatically detect and onboard cameras without manual configuration.

**Bounding Box** — A rectangular region drawn around a detected object in a video frame, defined by coordinates `(x, y, width, height)` along with a class label and confidence score.

**Embedding** — A dense vector representation of an input (e.g., a face crop). Embeddings enable similarity search — two images of the same person will produce vectors that are close together in the embedding space.

**NMS (Non-Maximum Suppression)** — A post-processing step that removes duplicate detections. When a model produces multiple overlapping bounding boxes for the same object, NMS keeps only the highest-confidence detection.

**IoU (Intersection over Union)** — A metric that measures the overlap between two bounding boxes. An IoU of 1.0 means perfect overlap; values above 0.5 are typically considered a valid match.

**FPS (Frames Per Second)** — The throughput of the video processing pipeline. Real-time processing typically requires 25–30 FPS per stream.

**Tensor** — A multi-dimensional array that serves as the fundamental data structure in deep learning. An image frame is typically represented as a tensor with shape `(batch, channels, height, width)`.

**Batch Size** — The number of frames processed simultaneously in a single inference call. Larger batches improve GPU utilisation but increase latency.

**TensorRT** — NVIDIA's deep learning inference optimiser and runtime. It takes trained models and optimises them for specific GPU hardware through layer fusion, precision calibration (FP16/INT8), and kernel auto-tuning.

**ONNX (Open Neural Network Exchange)** — An open format for representing machine learning models. ONNX allows models trained in PyTorch, TensorFlow, or other frameworks to be exported and run on different inference runtimes.

**Model Repository** — A structured directory that NVIDIA Triton Inference Server uses to load and serve models. Each model has its own directory with a configuration file and versioned model artifacts.

**DeepStream** — NVIDIA's streaming analytics toolkit built on GStreamer. It provides GPU-accelerated plugins for video decode, inference, tracking, and metadata handling in a pipeline architecture.

## The Video Ingestion Pipeline

Every computer vision platform begins with a fundamental problem: getting video frames from cameras into the inference engine as efficiently as possible. This is the **ingestion pipeline**, and getting it right has a massive impact on overall system performance.

![Video Ingestion Pipeline](/blog/deep-dive-real-time-computer-vision/inline-2.png)

### Camera Discovery with ONVIF

In production environments, you rarely hardcode camera URLs. Instead, you use ONVIF to discover cameras on the network automatically. ONVIF provides a WS-Discovery mechanism that broadcasts probe messages and collects responses from compatible devices.

```python
from onvif import ONVIFCamera

# Connect to an ONVIF-compatible camera
camera = ONVIFCamera('192.168.1.100', 80, 'admin', 'password')

# Get device information
device_info = camera.devicemgmt.GetDeviceInformation()
print(f"Manufacturer: {device_info.Manufacturer}")
print(f"Model: {device_info.Model}")

# Get the media service and stream URI
media_service = camera.create_media_service()
profiles = media_service.GetProfiles()

stream_setup = {
    'Stream': 'RTP-Unicast',
    'Transport': {'Protocol': 'RTSP'}
}

uri = media_service.GetStreamUri({
    'StreamSetup': stream_setup,
    'ProfileToken': profiles[0].token
})

print(f"RTSP URI: {uri.Uri}")
```

### RTSP Stream Relay with mediamtx

Rather than having every consumer connect directly to cameras (which strains the camera and creates a single point of failure), a media relay server like **mediamtx** sits between cameras and consumers. It accepts RTSP streams and republishes them, supporting protocol conversion to HLS or WebRTC for browser-based viewing.

```yaml
# mediamtx.yml
paths:
  camera-lobby:
    source: rtsp://192.168.1.100:554/stream1
    sourceProtocol: tcp
    readUser: viewer
    readPass: secure-password
    
  camera-entrance:
    source: rtsp://192.168.1.101:554/stream1
    sourceProtocol: tcp
    
  # Simulated stream for development/testing
  test-stream:
    runOnInit: >
      ffmpeg -re -stream_loop -1
      -i /data/test-video.mp4
      -c copy -f rtsp rtsp://localhost:$RTSP_PORT/$MTX_PATH
    runOnInitRestart: yes
```

### Frame Extraction with ffmpeg and OpenCV

Once the stream is available, we need to extract individual frames for inference. Here is a Python example using OpenCV with GPU-accelerated decoding:

```python
import cv2
import numpy as np

class FrameExtractor:
    def __init__(self, rtsp_url: str, target_fps: int = 10):
        self.cap = cv2.VideoCapture(rtsp_url, cv2.CAP_FFMPEG)
        self.cap.set(cv2.CAP_PROP_BUFFERSIZE, 3)
        self.target_fps = target_fps
        self.source_fps = self.cap.get(cv2.CAP_PROP_FPS) or 25.0
        self.skip_ratio = max(1, int(self.source_fps / target_fps))
        self.frame_count = 0
    
    def read_frame(self):
        """Read and yield frames at the target FPS."""
        while True:
            ret, frame = self.cap.read()
            if not ret:
                self.cap.set(cv2.CAP_PROP_POS_FRAMES, 0)
                continue
            
            self.frame_count += 1
            if self.frame_count % self.skip_ratio != 0:
                continue
            
            # Resize for inference (640x640 for YOLO)
            resized = cv2.resize(frame, (640, 640))
            
            # Normalise to float32 [0, 1]
            normalised = resized.astype(np.float32) / 255.0
            
            # Convert HWC to CHW (channels first)
            tensor = np.transpose(normalised, (2, 0, 1))
            
            yield tensor, frame  # Return both for overlay later
    
    def release(self):
        self.cap.release()
```

For production systems processing many streams, you would typically use **ffmpeg** with hardware-accelerated decoding (`-hwaccel cuda`) and pipe the raw frames directly to the inference engine, avoiding the overhead of Python-level frame handling entirely.

```bash
# Hardware-accelerated decode with NVIDIA GPU
ffmpeg -hwaccel cuda -hwaccel_output_format cuda \
  -rtsp_transport tcp \
  -i rtsp://mediamtx:8554/camera-lobby \
  -vf "fps=10,scale_cuda=640:640" \
  -f rawvideo -pix_fmt rgb24 \
  pipe:1
```

## GPU-Accelerated Inference with NVIDIA Triton

NVIDIA Triton Inference Server is the backbone of production computer vision inference. It is a purpose-built model serving platform that handles the complex orchestration of loading models, managing GPU memory, batching requests, and serving predictions over gRPC and HTTP.

![NVIDIA Triton Inference Server Architecture](/blog/deep-dive-real-time-computer-vision/inline-3.png)

### Model Repository Structure

Triton uses a structured directory layout called the **model repository**. Each model has its own directory containing a configuration file and versioned model artifacts:

```
model_repository/
├── yolov8_detection/
│   ├── config.pbtxt
│   └── 1/
│       └── model.plan          # TensorRT optimised
├── face_embedding/
│   ├── config.pbtxt
│   └── 1/
│       └── model.onnx          # ONNX format
├── classification/
│   ├── config.pbtxt
│   └── 1/
│       └── model.pt            # PyTorch TorchScript
└── ensemble_pipeline/
    └── config.pbtxt            # Chains models together
```

### Model Configuration

The `config.pbtxt` file defines how Triton loads and serves each model:

```protobuf
# yolov8_detection/config.pbtxt
name: "yolov8_detection"
platform: "tensorrt_plan"
max_batch_size: 16

input [
  {
    name: "images"
    data_type: TYPE_FP32
    dims: [ 3, 640, 640 ]
  }
]

output [
  {
    name: "output0"
    data_type: TYPE_FP32
    dims: [ 8400, 84 ]
  }
]

instance_group [
  {
    count: 2
    kind: KIND_GPU
    gpus: [ 0 ]
  }
]

dynamic_batching {
  preferred_batch_size: [ 4, 8, 16 ]
  max_queue_delay_microseconds: 5000
}

optimization {
  execution_accelerators {
    gpu_execution_accelerator: [
      { name: "tensorrt" }
    ]
  }
}
```

Key configuration elements:

1.  **`max_batch_size: 16`** — Triton will batch up to 16 inference requests together. This dramatically improves GPU throughput because GPUs are optimised for parallel computation.
2.  **`instance_group`** — Defines how many copies of the model to load. Running 2 instances on the same GPU allows Triton to overlap data transfer and computation.
3.  **`dynamic_batching`** — Triton collects incoming requests and batches them together. The `max_queue_delay_microseconds` parameter controls how long to wait before sending a partial batch — a critical latency vs throughput trade-off.

### Python Client for Inference

```python
import tritonclient.grpc as grpcclient
import numpy as np

class TritonDetector:
    def __init__(self, url: str = "localhost:8001"):
        self.client = grpcclient.InferenceServerClient(url=url)
        self.model_name = "yolov8_detection"
    
    def detect(self, frames: np.ndarray) -> list:
        """
        Run object detection on a batch of frames.
        frames: numpy array of shape (batch, 3, 640, 640)
        """
        inputs = [
            grpcclient.InferInput(
                "images", frames.shape, "FP32"
            )
        ]
        inputs[0].set_data_from_numpy(frames)
        
        outputs = [
            grpcclient.InferRequestedOutput("output0")
        ]
        
        result = self.client.infer(
            model_name=self.model_name,
            inputs=inputs,
            outputs=outputs,
        )
        
        raw_output = result.as_numpy("output0")
        return self._postprocess(raw_output)
    
    def _postprocess(self, raw: np.ndarray,
                     conf_threshold: float = 0.5,
                     iou_threshold: float = 0.45):
        """Apply NMS and filter low-confidence detections."""
        detections = []
        for batch_idx in range(raw.shape[0]):
            boxes = raw[batch_idx]
            # Extract coordinates, confidence, class scores
            cx, cy, w, h = (
                boxes[:, 0], boxes[:, 1],
                boxes[:, 2], boxes[:, 3]
            )
            class_scores = boxes[:, 4:]
            max_scores = np.max(class_scores, axis=1)
            class_ids = np.argmax(class_scores, axis=1)
            
            # Filter by confidence
            mask = max_scores > conf_threshold
            filtered_boxes = np.stack([
                cx[mask] - w[mask]/2,
                cy[mask] - h[mask]/2,
                cx[mask] + w[mask]/2,
                cy[mask] + h[mask]/2
            ], axis=1)
            
            # Apply NMS
            keep = self._nms(
                filtered_boxes,
                max_scores[mask],
                iou_threshold
            )
            
            detections.append({
                'boxes': filtered_boxes[keep],
                'scores': max_scores[mask][keep],
                'class_ids': class_ids[mask][keep]
            })
        
        return detections
    
    def _nms(self, boxes, scores, threshold):
        """Non-Maximum Suppression."""
        order = scores.argsort()[::-1]
        keep = []
        while order.size > 0:
            i = order[0]
            keep.append(i)
            if order.size == 1:
                break
            ious = self._compute_iou(
                boxes[i], boxes[order[1:]]
            )
            inds = np.where(ious <= threshold)[0]
            order = order[inds + 1]
        return keep
    
    def _compute_iou(self, box, boxes):
        """Compute IoU between one box and an array."""
        x1 = np.maximum(box[0], boxes[:, 0])
        y1 = np.maximum(box[1], boxes[:, 1])
        x2 = np.minimum(box[2], boxes[:, 2])
        y2 = np.minimum(box[3], boxes[:, 3])
        
        intersection = np.maximum(0, x2 - x1) * np.maximum(0, y2 - y1)
        area1 = (box[2] - box[0]) * (box[3] - box[1])
        area2 = (boxes[:, 2] - boxes[:, 0]) * (boxes[:, 3] - boxes[:, 1])
        union = area1 + area2 - intersection
        
        return intersection / (union + 1e-6)
```

### TensorRT Optimisation

For maximum inference throughput, models should be converted to TensorRT format. TensorRT applies layer fusion, precision calibration, and kernel selection for the specific GPU hardware:

```bash
# Convert ONNX model to TensorRT engine
trtexec \
  --onnx=yolov8n.onnx \
  --saveEngine=model.plan \
  --fp16 \
  --workspace=4096 \
  --minShapes=images:1x3x640x640 \
  --optShapes=images:8x3x640x640 \
  --maxShapes=images:16x3x640x640 \
  --verbose

# Benchmark the engine
trtexec --loadEngine=model.plan --batch=8 --iterations=1000
```

The `--fp16` flag enables half-precision inference, which typically doubles throughput with minimal accuracy loss. For even more aggressive optimisation, INT8 quantisation can be used with a calibration dataset.

## Building the DeepStream Analytics Pipeline

While you *could* stitch together ffmpeg, OpenCV, and Triton manually, NVIDIA's **DeepStream SDK** provides a production-grade, GStreamer-based framework that handles the entire video analytics pipeline on the GPU. Every stage — decode, inference, tracking, metadata handling, and output — runs as a GStreamer element with GPU acceleration.

![NVIDIA DeepStream Analytics Pipeline](/blog/deep-dive-real-time-computer-vision/inline-4.png)

### Pipeline Elements

A DeepStream pipeline is composed of GStreamer elements connected in sequence. Here are the critical ones:

| Element | Purpose | GPU Accelerated |
|---|---|---|
| `nvurisrcbin` | Source element for RTSP/file/USB inputs | ✓ Decode |
| `nvv4l2decoder` | Hardware video decoder (H.264/H.265) | ✓ |
| `nvstreammux` | Batches frames from multiple sources | ✓ |
| `nvinfer` | Runs inference (primary or secondary) | ✓ |
| `nvtracker` | Multi-object tracking (NvDCF, DeepSORT) | ✓ |
| `nvdsosd` | On-screen display (bounding boxes, labels) | ✓ |
| `nvmultistreamtiler` | Tiles multiple streams into grid view | ✓ |
| `nvmsgconv` | Converts metadata to JSON/protobuf | |
| `nvmsgbroker` | Publishes messages to Kafka/AMQP | |

### Flink Stream Processing

Apache Flink consumes from Kafka topics and performs real-time aggregation, anomaly detection, and alerting:

```java
// Flink job: Windowed occupancy aggregation
StreamExecutionEnvironment env =
    StreamExecutionEnvironment.getExecutionEnvironment();

KafkaSource<DetectionEvent> source = KafkaSource
    .<DetectionEvent>builder()
    .setBootstrapServers("kafka:9092")
    .setTopics("detection-events")
    .setGroupId("flink-occupancy")
    .setValueOnlyDeserializer(
        new DetectionEventDeserializer()
    )
    .build();

DataStream<DetectionEvent> events = env.fromSource(
    source,
    WatermarkStrategy
        .<DetectionEvent>forBoundedOutOfOrderness(
            Duration.ofSeconds(5)
        )
        .withTimestampAssigner(
            (event, ts) -> event.getTimestamp()
        ),
    "detection-source"
);

// Count unique tracked persons per camera per minute
DataStream<OccupancyCount> occupancy = events
    .flatMap(new ExtractPersonDetections())
    .keyBy(d -> d.getCameraId())
    .window(TumblingEventTimeWindows.of(
        Time.minutes(1)
    ))
    .aggregate(new UniqueTrackingIdCounter());

// Alert if occupancy exceeds threshold
DataStream<Alert> alerts = occupancy
    .filter(count -> count.getUniqueCount() > 50)
    .map(count -> new Alert(
        count.getCameraId(),
        "HIGH_OCCUPANCY",
        "Occupancy: " + count.getUniqueCount()
    ));

// Sink alerts back to Kafka
alerts.sinkTo(
    KafkaSink.<Alert>builder()
        .setBootstrapServers("kafka:9092")
```

## Security and Observability

A real-time computer vision platform handles sensitive data — video feeds, facial embeddings, tracking information. Security must be embedded at every layer, not bolted on as an afterthought.

### Defence in Depth

The security model follows a layered approach:

1.  **Perimeter** — VPN gateway restricts access to the infrastructure. Network Security Groups (NSGs) control inbound and outbound traffic at the VM level.
2.  **Identity** — Keycloak provides OIDC-based authentication with RBAC. Every API request carries a JWT token validated against Keycloak's public keys.
3.  **Secrets** — Vault issues short-lived dynamic credentials for database access. TLS certificates are distributed via shared volumes and rotated automatically.
4.  **Transport** — All inter-service communication uses TLS. Kafka and Flink use SSL for event transport. The reverse proxy terminates HTTPS for external clients.
5.  **Network** — Docker network segmentation ensures that inference services cannot directly reach the database, and event streaming services cannot access the identity layer.

### Edge-to-Cloud Synchronisation

For multi-site deployments, edge nodes run local inference with periodic model synchronisation from a central cloud repository. This ensures low-latency processing at the edge while maintaining consistent model versions across all sites:

```python
# Edge model sync agent
import boto3
import hashlib
import os

class ModelSyncAgent:
    def __init__(self, bucket: str, local_path: str):
        self.s3 = boto3.client('s3')
        self.bucket = bucket
        self.local_path = local_path
    
    def sync_models(self):
        """Pull latest models from central repository."""
        paginator = self.s3.get_paginator(
            'list_objects_v2'
        )
        
        for page in paginator.paginate(
            Bucket=self.bucket,
            Prefix='models/'
        ):
            for obj in page.get('Contents', []):
                key = obj['Key']
                local_file = os.path.join(
                    self.local_path,
                    key.replace('models/', '')
                )
                
                # Check if local file needs update
                if self._needs_update(key, local_file):
                    os.makedirs(
                        os.path.dirname(local_file),
                        exist_ok=True
                    )
                    self.s3.download_file(
                        self.bucket, key, local_file
                    )
                    print(f"Updated: {key}")
    
    def _needs_update(self, s3_key, local_path):
        if not os.path.exists(local_path):
            return True
        
        # Compare ETags
        response = self.s3.head_object(
            Bucket=self.bucket, Key=s3_key
        )
        remote_etag = response['ETag'].strip('"')
        
        with open(local_path, 'rb') as f:
            local_hash = hashlib.md5(
                f.read()
            ).hexdigest()
        
        return remote_etag != local_hash
```

## Cloud-Native Architecture on AWS

Everything we have covered so far — Docker Compose, manual GPU allocation, single-server deployment — works brilliantly for proof-of-concept and single-site installations. But when you need to support hundreds of customer edge sites, process millions of events per day, and deliver a multi-tenant SaaS portal with enterprise-grade security, you need a purpose-built cloud architecture.

### From Docker Networks to AWS Accounts

In the on-premise architecture, we used six Docker networks (`sec_network`, `api_network`, `inf_network`, `eve_network`, `bi_network`, `sup_network`) to segment traffic on a single server. In the cloud, these flat network boundaries are replaced entirely by **AWS account boundaries** and **VPC isolation** — a fundamentally different and much stronger security model. Here is how each on-premise concern maps to its cloud-native equivalent:

| On-Premise (Docker Network) | Cloud-Native (AWS) | Why It Changes |
|---|---|---|
| `sec_network` — Keycloak, Vault | **Identity Account** — Cognito user pools, Secrets Manager, WAF | Managed services replace self-hosted Keycloak and Vault. Cognito handles OAuth2/OIDC natively, and Secrets Manager provides automatic rotation without running your own Vault cluster. |
| `api_network` — REST API, Frontend, PostgreSQL | **Portal Account** — Next.js SSR (CloudFront + Lambda@Edge), API Gateway, Aurora Serverless v2 | The monolithic REST API and frontend move to a Next.js SSR application deployed via CloudFront with Lambda@Edge for server-side rendering and authentication. API routes replace the standalone REST API. PostgreSQL becomes Aurora Serverless v2 with automatic capacity scaling. |
| `inf_network` — Triton, DeepStream, mediamtx | **Edge devices** — IoT Greengrass components on NVIDIA Jetson/GPU servers | Inference stays on-premise (video never leaves the site), but Greengrass manages the lifecycle, updates, and monitoring of DeepStream and Triton. mediamtx is replaced by the Greengrass Stream Manager for media handling. |
| `eve_network` — Kafka, Flink, Schema Registry | **Edge Stream Platform Account** — MSK (managed Kafka), Managed Flink, Glue Schema Registry | Self-hosted Kafka and Flink are replaced by fully managed AWS equivalents. MSK handles broker management, patching, and scaling. Managed Flink removes the need to manage task managers. |
| `bi_network` — Superset dashboards | **Observability Account** — CloudWatch dashboards, Grafana on Fargate, QuickSight | Self-hosted Superset is replaced by a combination of CloudWatch (operational metrics), Grafana (custom dashboards), and optionally QuickSight (business analytics). |
| `sup_network` — Portainer, observability | **Management Account + Observability Account** — CloudWatch, CloudTrail, Systems Manager | Portainer is eliminated entirely — ECS provides native container management. Observability moves to CloudWatch cross-account with centralised CloudTrail in the Observability account. Systems Manager replaces SSH for edge device access. |

The key insight is that Docker network segmentation is a *software-level* boundary on a single host — a container escape or misconfigured network rule breaks the isolation. AWS account boundaries are *IAM-level* boundaries enforced by the AWS control plane itself. Even a root user in one account cannot access resources in another account unless explicit cross-account trust is configured. This is why the multi-account approach provides genuinely stronger isolation than the on-premise Docker network model.

In this section, we will go deep into the cloud-native architecture — account structure, edge-to-cloud communication, network design, application architecture, deployment pipelines, and cross-account event flow — each with practical examples and the reasoning behind every decision.

### Why Multi-Account?

Before diving into the architecture, it is worth understanding *why* we split the platform across multiple AWS accounts rather than deploying everything into a single account. There are five critical reasons:

1.  **Blast radius containment** — If a security incident occurs in the edge stream processing layer, it cannot affect the customer-facing portal or the identity system. Each account is an IAM boundary — credentials compromised in one account cannot access resources in another.
2.  **Independent scaling** — The edge stream platform might process 50 million events per day while the portal serves 500 concurrent users. These are fundamentally different scaling profiles that benefit from independent resource limits and cost tracking.
3.  **Regulatory compliance** — Many industries require separation of concerns for audit purposes. Having distinct accounts for identity, logging, and application workloads makes compliance evidence gathering straightforward.
4.  **Cost visibility** — AWS bills per account. With separate accounts, you immediately see that the edge stream platform costs $12,000/month while observability costs $800/month, without needing complex tagging strategies.
5.  **Team autonomy** — Different teams can own different accounts with their own CI/CD pipelines, IAM policies, and deployment cadences without stepping on each other's work.

### Multi-Account Strategy

Following the AWS Well-Architected Framework and the AWS Control Tower landing zone pattern, we structure the platform across seven dedicated accounts:

![Multi-Account AWS Architecture — Computer Vision Platform](/blog/deep-dive-real-time-computer-vision/inline-8.png)

Let us walk through each account in detail:

| Account | Purpose | Key Services | VPC CIDR |
|---|---|---|---|
| Management | Organisation root, billing, SSO | Organizations, IAM Identity Center, Cost Explorer | No VPC |
| Identity | Centralised authentication and user management | Cognito (user pools), WAF, Fargate (Identity API), CloudFront | `10.3.0.0/16` |
| Network | Shared networking hub, DNS, CDN, certificates | Transit Gateway, Direct Connect, Route 53, WAF, CloudFront, ACM, Parameter Store | `10.0.0.0/16` |
| Observability | Centralised logging, monitoring, dashboards | CloudWatch (cross-account), CloudTrail (org trail), Fargate (Grafana/SigNoz), CloudFront | `10.4.0.0/16` |
| Edge Stream Platform | Ingest and process edge device events | IoT Core, IoT Greengrass, MSK (Kafka), Flink, DynamoDB, Lambda, S3, EventBridge | `10.1.0.0/16` |
| Portal (dev/tst/prod) | Customer-facing web application and API | CloudFront, Lambda@Edge (Next.js SSR + Auth), S3 (static assets), API Gateway, Aurora Serverless v2, Secrets Manager, EventBridge | `10.2.0.0/16` |
| Delegate | Delegated admin for Identity Center | IAM Identity Center (Delegated Admin) | No VPC |

Each account has its own VPC with non-overlapping CIDRs connected through a **Transit Gateway** in the Network account. This ensures that inter-account traffic never traverses the public internet. The Management and Delegate accounts do not require VPCs as they only host IAM-level resources.

A critical design decision is the use of **AWS Organizations** with Service Control Policies (SCPs) to enforce guardrails across all accounts:

### Edge-to-Cloud with AWS IoT Greengrass

The most significant architectural shift from on-premise to cloud is how edge devices communicate with the platform. In the Docker Compose world, we used direct VPN tunnels and MQTT connections between the edge server and cloud services. In the cloud-native architecture, we use **AWS IoT Core** and **IoT Greengrass v2** to create a managed, secure, and scalable edge-to-cloud pipeline.

![Edge-to-Cloud — IoT Greengrass + DeepStream](/blog/deep-dive-real-time-computer-vision/inline-9.png)

The fundamental design principle here is that **video never leaves the customer site**. Camera streams are processed locally on GPU hardware (NVIDIA Jetson or a server with discrete GPUs), and only the resulting metadata — bounding boxes, tracking IDs, face embeddings, timestamps, and event classifications — is transmitted to the cloud over MQTT/TLS. This reduces bandwidth by orders of magnitude (megabytes of metadata vs gigabytes of video per hour) and addresses data sovereignty concerns.

Here is how the edge-to-cloud flow works, step by step:

1.  **Greengrass Core Device** — Each customer site runs a Greengrass v2 core device. This is either an NVIDIA Jetson (for smaller deployments with 10-20 cameras) or a rack-mounted GPU server (for larger sites with 100+ cameras). Greengrass provides a managed runtime for deploying, updating, and monitoring edge components.

2.  **DeepStream as a Greengrass Component** — The DeepStream inference pipeline runs as a Greengrass component. Greengrass manages its lifecycle — starting, stopping, monitoring health, and rolling back failed updates. The component recipe defines dependencies, configuration, and artifacts:

```yaml

ComponentPublisher: VisionPlatform

ComponentDependencies:
```

3.  **MQTT Message Publishing** — Detection events are published to IoT Core over MQTT/TLS with X.509 certificate-based authentication. Each edge device has a unique certificate provisioned during onboarding. Messages follow a topic hierarchy that enables fine-grained filtering:

```python
# Edge-side MQTT publishing
import json
import time
import paho.mqtt.client as mqtt
import ssl

class CloudPublisher:
    def __init__(self, thing_name: str, endpoint: str):
        self.client = mqtt.Client(
            client_id=thing_name
        )
        self.client.tls_set(
            ca_certs="/greengrass/v2/rootCA.pem",
            certfile=f"/greengrass/v2/thingCert.crt",
            keyfile=f"/greengrass/v2/privKey.key",
            tls_version=ssl.PROTOCOL_TLSv1_2,
        )
        self.client.connect(endpoint, 8883)
        self.client.loop_start()
        self.topic_prefix = f"vision/{thing_name}"

    def publish_detection(self, event: dict):
        """Publish detection event to IoT Core."""
        payload = {
            "site_id": event["site_id"],
            "camera_id": event["camera_id"],
            "timestamp": int(time.time() * 1000),
            "frame_number": event["frame_number"],
            "detections": [
                {
                    "class": d["class_name"],
                    "confidence": round(
                        d["confidence"], 3
                    ),
                    "tracking_id": d.get(
                        "tracking_id"
                    ),
                    "bbox": {
                        "x": round(d["x"], 1),
                        "y": round(d["y"], 1),
                        "w": round(d["w"], 1),
                        "h": round(d["h"], 1),
                    },
                }
                for d in event["detections"]
                if d["confidence"]
                > self.confidence_threshold
            ],
        }
        self.client.publish(
            f"{self.topic_prefix}/detections",
            json.dumps(payload),
            qos=1,
        )
```

### Network Architecture

The network layer is the connective tissue that holds the multi-account architecture together. Every VPC needs to communicate with specific other VPCs, and customer edge sites need secure connectivity into the cloud. Getting this right is essential for both security and performance.

![Network Architecture — Transit Gateway + Direct Connect](/blog/deep-dive-real-time-computer-vision/inline-10.png)

Let us walk through each networking component and why it exists:

1.  **Transit Gateway** — This is the central routing hub, deployed in the Network account and shared to all other accounts via AWS Resource Access Manager (RAM). It replaces the complexity of VPC peering (which is point-to-point and does not scale) with a hub-and-spoke model. Route tables on the Transit Gateway control which VPCs can communicate — for example, the Edge Stream Platform VPC can reach the Portal VPC (to forward events), but the Portal VPC cannot directly reach the Observability VPC (it sends logs via CloudWatch cross-account, not direct network connectivity).

2.  **Direct Connect** — For enterprise customers with high-volume sites (50+ cameras), we provision dedicated AWS Direct Connect circuits. A Direct Connect circuit provides a private, dedicated network connection from the customer's on-premise data centre (often co-located in an Equinix facility) into the AWS VPC. This avoids internet variability and provides consistent sub-10ms latency for metadata delivery. The connection terminates in the Network account VPC and routes through the Transit Gateway to the Edge Stream Platform account.

3.  **Site-to-Site VPN** — For smaller customer sites or as a backup for Direct Connect, we use encrypted IPSec VPN tunnels. These can be provisioned in minutes (compared to weeks for Direct Connect) and provide adequate bandwidth for metadata-only traffic.

4.  **VPC Endpoints** — This is often overlooked but critically important. Every account deploys VPC endpoints for the AWS services it uses (S3, DynamoDB, Secrets Manager, ECR, CloudWatch Logs, STS). Without VPC endpoints, API calls to AWS services would need to traverse the internet via a NAT Gateway, adding latency and cost, and creating a potential data exfiltration path. With VPC endpoints, all AWS API traffic stays within the AWS backbone network.

5.  **DNS (Route 53)** — The Network account hosts Route 53 private hosted zones that are shared with other accounts. This provides consistent internal DNS resolution across all VPCs. Public DNS for customer-facing endpoints (the portal, the API) is managed through Route 53 public hosted zones with DNSSEC enabled.

### Portal Application Architecture

The customer-facing portal is where all the processed intelligence becomes actionable. Users log in, view real-time detections, configure alerts, manage devices, and access dashboards. Rather than running containers on Fargate, the portal is built as a **Next.js SSR application** deployed through CloudFront with Lambda@Edge — a fully serverless architecture that eliminates the need to manage any compute infrastructure.

![Portal Account — Application Architecture](/blog/deep-dive-real-time-computer-vision/inline-11.png)

The key architectural decision here is using **Next.js server-side rendering at the edge** via Lambda@Edge (or CloudFront Functions for lighter work). This gives us the developer experience of a modern React framework — file-based routing, server components, API routes, middleware — with the performance of edge-deployed serverless compute. There are no containers to manage, no ALBs to configure, and no auto-scaling policies to tune.

The request flow from a user's browser to the backend follows a carefully layered path:

1.  **CloudFront** — The first point of contact. CloudFront distributes the application globally with ACM-managed TLS certificates. Cache behaviours are configured per path pattern: `/_next/static/*` serves pre-built assets directly from S3 with immutable caching (1 year TTL), `/_next/image/*` routes to the image optimisation Lambda, and all other requests (`/*`) are forwarded to the Next.js SSR Lambda@Edge function. This means static assets are served from the nearest edge location with zero origin latency.

2.  **Lambda@Edge (Authentication)** — A Viewer Request function intercepts every incoming request before it reaches the origin. It validates the JWT token from the `access_token` cookie against the Identity account's Cognito user pool. If the token is missing, expired, or invalid, the function returns a 302 redirect to the Cognito hosted login page. If valid, it adds decoded user claims as custom headers (`X-User-Id`, `X-User-Role`) and forwards the request. This means unauthenticated requests are rejected *at the edge* — they never reach our application code.

3.  **Lambda@Edge (Next.js SSR)** — An Origin Request function runs the Next.js server-side rendering logic. This handles dynamic page rendering, API route execution, server-side data fetching, and middleware. The function has access to environment variables for database connection strings (via Secrets Manager) and EventBridge bus ARNs. For API routes that need database access, the Lambda connects to Aurora through the VPC via a Lambda VPC configuration.

4.  **S3 (Static Assets)** — The Next.js build output is split into static assets (JS bundles, CSS, images, pre-rendered HTML) stored in S3, and server-side code packaged into the Lambda function. The `next build` output includes a manifest that CloudFront uses to determine which requests can be served from S3 vs which need Lambda processing.

5.  **Aurora Serverless v2** — PostgreSQL for relational data (users, devices, event records, configuration). Serverless v2 auto-scales capacity between a minimum and maximum ACU (Aurora Capacity Unit), which is ideal for bursty workloads where event volume spikes during business hours and drops overnight. The Next.js API routes connect to Aurora through the VPC — Lambda@Edge functions are configured with VPC access for database queries.

6.  **API Gateway WebSocket** — Real-time updates flow from the Edge Stream Platform account through EventBridge (cross-account), are processed by Lambda, and pushed to connected clients via API Gateway WebSocket connections. This provides the live detection feed in the portal UI without polling. The Next.js client-side code connects to the WebSocket endpoint on page load.

7.  **EventBridge** — Cross-account events from the Edge Stream Platform are consumed by event rules that trigger Lambda functions for database writes, alert processing, and WebSocket push notifications.

> **Note:** For teams that prefer a container-based approach or need long-running server processes (e.g., WebSocket servers, background workers), the Next.js application can alternatively be deployed on **ECS Fargate** behind an ALB. This trades the zero-infrastructure benefit of Lambda@Edge for more control over the runtime environment, persistent connections, and simpler local development parity. Both approaches work well — the serverless path is preferred for cost efficiency and operational simplicity.

```typescript
// lib/lambda/edge-auth/index.ts
import {
  CloudFrontRequestEvent,
  CloudFrontRequestResult,
} from "aws-lambda";
import { CognitoJwtVerifier } from "aws-jwt-verify";

const verifier = CognitoJwtVerifier.create({
  userPoolId: process.env.USER_POOL_ID!,
  clientId: process.env.CLIENT_ID!,
  tokenUse: "access",
});

const LOGIN_URL = process.env.LOGIN_URL!;
const PUBLIC_PATHS = [
  "/health",
  "/favicon.ico",
  "/_next/static/",
  "/_next/image/",
  "/api/auth/",
];

export async function handler(
  event: CloudFrontRequestEvent
): Promise<CloudFrontRequestResult> {
  const request = event.Records[0].cf.request;
  const uri = request.uri;

  // Allow public paths without auth
  if (PUBLIC_PATHS.some((p) => uri.startsWith(p))) {
    return request;
  }

  // Extract token from cookie or Authorization header
  const cookies = request.headers.cookie ?? [];
  const cookieStr = cookies
    .map((c) => c.value)
    .join("; ");
  const tokenMatch = cookieStr.match(
    /access_token=([^;]+)/
  );
  const authHeader =
    request.headers.authorization?.[0]?.value;
  const token =
    tokenMatch?.[1] ??
    authHeader?.replace("Bearer ", "");

  if (!token) {
    return redirectToLogin(uri);
  }

  try {
    const payload = await verifier.verify(token);
    // Add user claims as headers for downstream
    request.headers["x-user-id"] = [
      { key: "X-User-Id", value: payload.sub },
    ];
    request.headers["x-user-email"] = [
      {
        key: "X-User-Email",
        value: (payload as Record<string, string>)
          .email ?? "",
      },
    ];
    return request;
  } catch {
    return redirectToLogin(uri);
  }
}

function redirectToLogin(
  returnPath: string
): CloudFrontRequestResult {
  return {
    status: "302",
    statusDescription: "Found",
    headers: {
      location: [
        {
          key: "Location",
          value: `${LOGIN_URL}?redirect=${encodeURIComponent(returnPath)}`,
        },
      ],
    },
  };
}
```

### Cross-Account Event Flow

One of the most critical patterns in the multi-account architecture is how detection events flow from the Edge Stream Platform account to the Portal account. We use **EventBridge cross-account event buses** for this, which provides loose coupling, built-in retry with dead-letter queues, content-based filtering, and zero infrastructure to manage.

The flow works as follows:

1.  Detection events arrive in the Edge Stream account via IoT Core → MSK.
2.  A Flink job processes the raw events (aggregation, enrichment, deduplication).
3.  Processed events are published to the local EventBridge bus with a custom source (`vision.edge-stream`).
4.  An EventBridge rule forwards matching events to the Portal account's event bus (cross-account).
5.  The Portal account routes events to Lambda (for database writes), SQS (for reliable processing), and API Gateway WebSocket (for real-time UI).

```typescript

  eventPattern: {
    source: ["vision.edge-stream"],
    detailType: [
      "DetectionEvent",
      "AlertEvent",
      "OccupancyUpdate",
    ],
  },

    new targets.EventBus(portalEventBus),
```

```typescript
});

// Route detection events to Lambda processor
new events.Rule(this, "ProcessDetections", {
  eventBus,
  eventPattern: {
    source: ["vision.edge-stream"],
    detailType: ["DetectionEvent"],
  },
  targets: [
    new targets.LambdaFunction(processorFn, {
      deadLetterQueue: dlq,
      retryAttempts: 2,
    }),
  ],
});

// Route alerts to SNS for push notifications
new events.Rule(this, "ProcessAlerts", {
  eventBus,
  eventPattern: {
    source: ["vision.edge-stream"],
    detailType: ["AlertEvent"],
  },
  targets: [
    new targets.SnsTopic(alertTopic),
    new targets.LambdaFunction(webhookFn),
  ],
});
```

This pattern ensures loose coupling between accounts — the Edge Stream account does not need to know anything about the Portal account's internal architecture. Events are published to the bus, and the Portal account decides how to consume them. If the Portal account adds a new consumer (say, a data warehouse ingestion pipeline), no changes are needed in the Edge Stream account — it just adds a new EventBridge rule.

Don't worry if all of this seems overwhelming. Building a real-time computer vision platform is genuinely complex — it spans video engineering, GPU computing, distributed systems, security, and infrastructure. Each layer we covered — ingestion, inference, streaming, edge deployment, and cloud-native architecture — can be improved independently as your requirements grow. The move from on-premise Docker Compose to a multi-account AWS architecture is not a single leap; it is a series of well-planned incremental steps, each one reducing operational burden while increasing scalability and security.
